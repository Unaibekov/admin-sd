# Admin Context

## Роль админки
- Админка не является источником истины.
- Админка получает ZIP-архив от мобильного приложения.
- Админка предназначена для просмотра, проверки и анализа отчетов.

## Текущий сценарий
1. Мобильное приложение формирует ZIP.
2. В ZIP лежит `report.json` и папка `photos/`.
3. Админка принимает ZIP.
4. Админка извлекает и показывает отчет.

## Что админка должна уметь
- Просматривать партии и их карточки.
- Открывать журнал операций.
- Смотреть фотографии по операциям и стартовым данным.
- Понимать текущий остаток и статусы партии.
- Показывать summary по всему отчету.

## Структура данных для админки
- Корень отчета: `reportId`, `createdAt`, `appVersion`, `deviceId`, `user`, `summary`, `cards`.
- Карточка: `cardId`, паспорт, статус, количество, местоположение, события, `extraFields`.
- Событие: `eventId`, тип, дата, фото, комментарий, детали, `extraFields`.

## Ограничения
- Админка не должна пересчитывать источник истины поверх мобильного приложения.
- Админка не должна менять формат `report.json` без согласования.
- Админка не должна ожидать скрытые поля, которых нет в экспорте.
- Админка должна работать с тем, что реально отдает мобильное приложение, а не с гипотетической серверной моделью.

## Принцип данных
- Истина создается на мобильном устройстве.
- Админка только читает и отображает.
- Исправления и внесение событий должны происходить в мобильном приложении, если для этого не появится отдельный backend-контур.

## Будущее развитие
- Поиск по партиям и операциям.
- Просмотр фото встраиваемым архивом.
- Фильтры по стадиям, статусам и рискам.
- Сводные показатели по партии, пользователю, устройству и периоду.
- Импорт отчетов без потери полей `extraFields`.
