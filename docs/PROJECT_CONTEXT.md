# Project Context

## Назначение Sadovnik Diary
Sadovnik Diary - мобильный журнал партий растений и посадочного материала. Приложение ведет карточки партий по стадиям выращивания, хранит журнал операций, показывает календарь действий, строит задачи по уходу и умеет выгружать отчеты для передачи во внешнюю админку.

## Целевая аудитория
- Операторы, которые создают партии, фиксируют события и прикладывают фото.
- Агрономы, которые контролируют карантин, проблемы и переходы между стадиями.
- Администраторы и суперадминистраторы, которым нужен доступ к отчетам, корректировкам и служебным действиям.

## Что является источником истины
- Источник истины сейчас - мобильное приложение.
- Карточки хранятся локально в `AsyncStorage`.
- Для будущей админки приложение формирует ZIP-отчет с `report.json` и фотографиями.
- Внешняя веб-админка не является источником истины, она только потребляет отчет.

## Основные сущности
- `Batch` или карточка партии.
- `Operation` или запись журнала.
- `Task` или задача по уходу.
- `Report` или экспортируемый отчет.
- `User` или текущий пользователь/профиль сотрудника.
- `Photo` как набор URI и файлов в экспорте.
- `Location` как текстовое описание местоположения и данные перемещения.

## Жизненный цикл партии
1. Партия создается в стадии `Введение в культуру`.
2. На этапе введения фиксируется создание партии и QR-состояние.
3. Далее партия может переходить в `Клонирование`, `Адаптацию`, `Теплицу`, `Закалку`, `Высадку`.
4. На каждой стадии в журнал добавляются операции с датой, фото и атрибутами события.
5. Стадия, остаток и статус партии вычисляются из карточки и журнала, а не хранятся только как одно поле.
6. Для ухода строятся задачи по расписанию.
7. Для внешней системы формируется ZIP-архив с `report.json`.

## Стадии выращивания
- `Введение в культуру`
- `Клонирование`
- `Адаптация`
- `Теплица`
- `Закалка`
- `Высадка`

## Текущий MVP
- React Native + Expo, JavaScript.
- Локальная работа без обязательного бэкенда.
- Авторизация через логин/пароль, PIN и биометрию.
- Создание и редактирование карточек.
- Журнал операций по датам.
- Календарь и задачи по уходу.
- Фото как доказательства действий.
- QR-сценарий для партий.
- Экспорт Excel и ZIP-отчета.

## Роль будущей админки
- Принимать ZIP-отчет.
- Просматривать `report.json`.
- Служить системой просмотра отчетов.
- Не менять модель как источник истины.

## Требования к дальнейшей разработке
- Не вводить новые сущности без явной необходимости.
- Не менять формат отчета без отдельного решения.
- Не ломать обратную совместимость локальных данных.
- Считать документы в `docs/` источником истины для новых задач Codex.
