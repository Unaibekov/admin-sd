# Business Rules

## 1. Создание и сохранение партий
- Новая карточка создается как партия со стадией, количеством, кодом, источником материала, родительской партией, фото и статусом стерильности.
- При создании автоматически добавляется операция `batchCreated`.
- Для интро-потока дополнительно фиксируется `qrGenerated`.
- При редактировании карточки операции сохраняются, а не пересоздаются целиком.

## 2. QR-логика
- `qrStatus` вычисляется как `none`, `pending_print` или `printed`.
- Если код уже есть и QR не печатался, состояние считается `pending_print`.
- Если QR уже распечатан, редактирование идентификационных полей ограничивается для обычных ролей.
- После печати QR поля идентичности можно менять только администратору или суперадминистратору.

## 3. Статусы партии
- Базовые статусы: `active`, `quarantine`, `partial`, `problem`, `sold`, `archived`.
- Статус может пересчитываться по операциям и текущему остатку.
- `quarantine` имеет приоритет над обычным активным состоянием.
- Если остаток стал нулевым после продажи, партия считается проданной.
- В некоторых сценариях итоговый `status` карточки становится `archived`.

## 4. Логика количества
- Количество в карточке - это стартовый объем партии.
- Текущее количество считается из карточки и операций.
- `sale`, `death`, `discard`, `introLoss` уменьшают остаток.
- `propagation` увеличивает остаток.
- `statusChange` может содержать несколько счетчиков, влияющих на остаток.
- Операция не может списать больше, чем текущий остаток.

## 5. Карантин и проблемы
- Перевод в карантин требует причины.
- Снятие карантина разрешено только ролям `agronomist`, `admin`, `superadmin`.
- Снятие карантина доступно только если партия действительно находится в карантине, если это не режим редактирования.
- Операция `problem` может переводить партию в `quarantine` или `problem` в зависимости от типа проблемы.
- Наличие `contamination` автоматически помечает стерильность как `contaminated`.

## 6. Переходы между стадиями
- Переход возможен только вперед по последовательности стадий.
- Для перехода из `Введение в культуру` нужна активная партия и сгенерированный QR.
- Для перехода из `Клонирование` нужны укоренившиеся растения и остаток больше нуля.
- Для перехода из `Адаптация` нужен не критический стресс и остаток больше нуля.
- Партия с заражением не может свободно двигаться по стадиям, пока ограничение не снято.

## 7. Правила по календарю
- Производственные события по статусам можно фиксировать только на текущую дату.
- Календарь используется как точка входа в журнал событий по выбранной дате.
- Для стадии `stageChange` дата перехода сохраняется отдельно от даты создания операции.

## 8. Логика задач
- Задачи строятся только для стадий `Адаптация`, `Теплица`, `Закалка`.
- Отмененные, архивные и проданные партии в задачи не попадают.
- Задача появляется по ближайшей дате следующего ухода.
- Просроченные задачи сортируются выше обычных.

## 9. Логика журнала
- В журнал попадают операции карточки, кроме служебных `stageSettingsUpdated`.
- Важные события включают проблемы, карантин, потери, стадии и критические риски.
- Переходы стадий группируются отдельно от обычных действий.

## 10. Экспорт
- Excel-отчет строится из карточек и журнала.
- ZIP-отчет содержит `report.json` и файлы фото.
- `report.json` не должен меняться без необходимости, потому что это контракт для будущей админки.
