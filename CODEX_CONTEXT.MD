# Контекст для Codex

Работа ведется над проектом веб-админки Sadovnik Admin.

Корневая папка проектов:

```text
C:\projects
```

Внутри находятся два связанных проекта:

```text
C:\projects\SadovnikDiary3
```

Основное мобильное приложение Sadovnik Diary.

И

```text
C:\projects\adminSD
```

Веб-админка Sadovnik Admin.

==================================================
ИСТОЧНИК ИСТИНЫ
===============

Главный проект:

```text
C:\projects\SadovnikDiary3
```

Именно в нем находится актуальная бизнес-логика системы.

В SadovnikDiary3 содержатся:

* стадии выращивания;
* операции;
* структура партий;
* правила перехода между стадиями;
* правила расчета количества;
* журнал событий;
* задачи;
* экспорт отчетов;
* структура report.json;
* бизнес-правила проекта.

Если между логикой админки и мобильного приложения возникает конфликт:

приоритет всегда имеет SadovnikDiary3.

==================================================
ОБЯЗАТЕЛЬНОЕ ИЗУЧЕНИЕ ПЕРЕД ЛЮБОЙ ЗАДАЧЕЙ
=========================================

Перед выполнением любой задачи по проекту adminSD необходимо изучить проект:

```text
C:\projects\SadovnikDiary3
```

В первую очередь изучать:

```text
C:\projects\SadovnikDiary3\docs
```

Особенно:

* PROJECT_CONTEXT.md
* BUSINESS_RULES.md
* DATA_MODEL.md
* STAGES_AND_OPERATIONS.md
* REPORT_FORMAT.md
* ADMIN_CONTEXT.md
* DEVELOPMENT_RULES.md

Также изучать реализацию:

```text
C:\projects\SadovnikDiary3\src\domain
C:\projects\SadovnikDiary3\src\services
C:\projects\SadovnikDiary3\src\screens
```

Особое внимание:

* batch.js
* appDerivedState.js
* tasks.js
* shareZipReportService.js
* экран журнала
* экран карточки партии
* экран отчетов

==================================================
ЗАЧЕМ НУЖНО ИЗУЧАТЬ МОБИЛЬНОЕ ПРИЛОЖЕНИЕ
========================================

Админка должна отображать данные так же, как они существуют в мобильном приложении.

Нельзя:

* придумывать новые стадии;
* придумывать новые статусы партий;
* придумывать новые типы операций;
* придумывать новые поля report.json;
* придумывать новую терминологию.

Нужно использовать существующие сущности из SadovnikDiary3.

==================================================
ОГРАНИЧЕНИЯ
===========

Проект:

```text
C:\projects\SadovnikDiary3
```

используется только как reference.

Запрещено:

* изменять файлы SadovnikDiary3;
* делать коммиты в SadovnikDiary3;
* выполнять рефакторинг SadovnikDiary3;
* исправлять код SadovnikDiary3.

Разрешено:

* читать код;
* читать документацию;
* использовать структуры данных;
* использовать report.json;
* использовать бизнес-правила;
* использовать терминологию проекта.

==================================================
ПРАВИЛА ДЛЯ ПРОЕКТА adminSD
===========================

Проект:

```text
C:\projects\adminSD
```

является read-only системой.

Админка должна:

* принимать ZIP-отчеты;
* читать report.json;
* отображать партии;
* отображать стадии;
* отображать журнал;
* отображать проблемы;
* отображать потери;
* отображать продажи;
* отображать фотографии;
* строить аналитику по данным отчета.

Админка НЕ должна:

* редактировать партии;
* изменять стадии;
* менять количество растений;
* изменять report.json;
* отправлять изменения обратно в мобильное приложение.

==================================================
ПЕРЕД ВЫПОЛНЕНИЕМ КАЖДОЙ ЗАДАЧИ
===============================

Перед началом работы показать:

1. Какие документы из SadovnikDiary3 были изучены.
2. Какие файлы SadovnikDiary3 использовались как reference.
3. Какие файлы adminSD будут изменены.
4. Какие сущности используются из мобильного приложения.
5. Есть ли конфликт между новой задачей и существующей логикой SadovnikDiary3.

Если найден конфликт между логикой adminSD и SadovnikDiary3:

сначала описать конфликт и предложить решение, не приступая к реализации.

==================================================
ГЛАВНЫЙ ПРИНЦИП
===============

SadovnikDiary3 — источник истины.

adminSD — инструмент просмотра и анализа данных.

Любая логика, реализуемая в adminSD, должна соответствовать структурам данных и бизнес-правилам проекта SadovnikDiary3.
